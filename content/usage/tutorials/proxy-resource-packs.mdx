# 🌐 Resource packs on a proxy

This guide explains how to distribute Oraxen resource packs across a Velocity or BungeeCord network without unnecessary pack downloads.

## Understand the limitation first

Minecraft does not automatically combine the resource packs sent by different backend servers. A proxy plugin such as [PackLayer](/compatibility/packlayer) can prevent a duplicate send, but only when the **final pack is identical** to the one the player already has.

`multi_version_packs` solves a different problem: it selects packs for different **Minecraft client versions**. It does not make different backend-server packs interchangeable. See [ViaVersion compatibility](/compatibility/viaversion) when client-version support is the goal.

## Choose the correct setup

| Requirement | Recommended setup | Expected player experience |
| --- | --- | --- |
| Every backend uses the same Oraxen assets | Use one identical generated pack on every backend and install PackLayer on the proxy. | One download, then seamless switching. |
| Servers need different features, but all assets can live in one pack | Generate one shared pack and send it from the lobby or entry server only. | One download when entering the network. |
| Servers require genuinely different packs and players use Minecraft 1.20.3+ | Use distinct resource-pack layers. | Each distinct layer may still need to download once. |
| Servers require genuinely different packs on older clients | Let each server send its own pack. | A pack change or reload on server switch is expected. |

## Option A: one identical pack on every backend

Use this when your lobby, survival, minigame, and other servers should all use the same Oraxen assets.

<Steps>

### Install PackLayer on the proxy

Install PackLayer in the `plugins` directory of the **Velocity or BungeeCord proxy**. Do not install it on the backend Paper servers.

Keep its default `HASH_ONLY` skip mode unless you have a specific reason to use another mode. Hash checking matters especially when self-hosting, because a pack URL can stay the same after its contents change.

### Keep the Oraxen pack source identical

The simplest reliable approach is to copy the complete `plugins/Oraxen` directory from one backend to every backend that should share the pack.

This includes item files, resource-pack models, textures, fonts, sounds, merged content, and settings. Copying only `settings.yml` is not enough.

### Regenerate the pack on every backend

Restart the servers or run:

```text
/oraxen reload all
```

Wait until each server has finished generating and uploading its pack before testing. A player who joins while generation or uploading is still in progress can receive no pack or an older one.

### Verify the actual generated pack

Check each backend's console output or configured pack host. The generated pack must have the same final hash on every backend. An identical URL is a useful quick check with hosts that produce content-specific URLs, but the SHA-1 hash is the source of truth.

### Test through the proxy

Join the first backend, wait until the pack is fully loaded, then switch servers through the proxy. Enable PackLayer's `debug: true` setting if a pack is sent again unexpectedly.

</Steps>

<Callout type="warning">
  PackLayer skips a duplicate pack; it does not merge different packs. A single changed texture, model, font, sound, or generated definition makes the final packs different.
</Callout>

## Option B: send one shared pack from the entry server

Use this when backends have different gameplay, but all required client assets can be included in one common pack.

1. Put the union of required Oraxen assets into the pack generated by the lobby or entry server.
2. Keep resource-pack dispatch enabled there.
3. Disable dispatch on the remaining backend servers.
4. Keep the matching Oraxen item and mechanic configuration on every backend that uses those items or blocks.

Lobby or entry server:

```yaml
Pack:
  dispatch:
    send: true
```

Backend servers:

```yaml
Pack:
  dispatch:
    send: false
```

This assumes players enter through the server that sends the shared pack. Players who can connect directly to a backend need another reliable way to receive the pack.

## Option C: use resource-pack layers

On Minecraft `1.20.3+`, Oraxen can add a pack instead of replacing the active one. Give each independently dispatched pack a stable, distinct layer name:

```yaml
Pack:
  dispatch:
    layer: "oraxen-survival"
```

For another server, use a different name, for example `oraxen-minigames`.

Layers are useful when separate packs are necessary, but they do not remove the initial download for each unique pack. They are not a replacement for one shared pack plus PackLayer.

<Callout type="info">
  A non-empty `Pack.dispatch.layer` makes Oraxen use the layered resource-pack API. This is intended for BungeeCord or Velocity environments and requires Minecraft `1.20.3+`.
</Callout>

## Troubleshooting checklist

- Confirm PackLayer is installed on the proxy, not a backend server.
- Compare the final pack hashes, not just the names of Oraxen configuration files.
- Check that every backend finished pack generation and uploading before a player joined.
- Ensure the backend is not excluded by PackLayer's server filter.
- When self-hosting, avoid URL-only duplicate detection because the URL can remain unchanged after a new upload.
- Use `debug: true` in PackLayer to see why it did or did not skip a send.
