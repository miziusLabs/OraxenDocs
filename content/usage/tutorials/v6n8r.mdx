# 🌐 Resource packs on a proxy

## Overview

This tutorial explains how to use Oraxen resource packs on a Velocity or BungeeCord network without showing players the same resource-pack request every time they switch servers.

OneTimePack is a proxy plugin for Velocity and BungeeCord that filters duplicate resource-pack sends. It only suppresses a pack when it matches the pack the player already has.

<Callout type="info">
  OneTimePack does not merge different resource packs. A backend still needs to send its pack when its resources differ from the one already loaded by the player.
</Callout>

## Installation

<Steps>

### Install OneTimePack on the proxy

Install OneTimePack in the `plugins` folder of your Velocity or BungeeCord proxy. Do not install it on backend Paper servers.

### Restart the proxy

Restart the proxy once so OneTimePack can create its configuration files.

### Keep shared resource packs identical

Every backend server that should share a pack must generate the same final resource pack. Keep the Oraxen item configuration, models, textures, fonts, sounds, and other pack content in sync between those servers.

Copying only `settings.yml` is not enough. A changed model, texture, font, sound, or generated definition makes the final packs different.

### Use the default comparator

OneTimePack compares available resource-pack UUIDs, hashes, and URLs by default.

```yaml
protocol:
  default:
    comparator: !UUID OR !HASH OR URL
```

The `!` prefix means that a UUID or hash is only compared when it exists.

### Test server switching

Join one backend through the proxy and wait until the resource pack has loaded. Then switch to another backend that uses the same final pack.

The resource-pack request should not appear again. If it does, compare the final packs generated by both servers and enable OneTimePack debug logging.

</Steps>

## Different packs on different servers

OneTimePack only suppresses a duplicate resource pack. A backend with a different pack must still send that pack to the player.

On Minecraft `1.20.3+`, Oraxen can use resource-pack layers to keep distinct packs active together. Give each independently dispatched pack a unique layer name.

```yaml
Pack:
  dispatch:
    layer: "oraxen-survival"
```

Layers allow multiple packs to coexist, but players still have to download each unique pack at least once.

## Troubleshooting

**The resource-pack request appears again after switching servers**

Confirm that OneTimePack is installed on the proxy and that both backends generate the same final pack. Set `log-level` under OneTimePack's `plugin` section to `4`, then restart the proxy to enable debug messages.

```yaml
plugin:
  log-level: 4
```

**Players receive no pack after a restart**

OneTimePack filters pack sends; it does not send Oraxen's pack itself. Confirm that pack dispatch remains enabled in Oraxen and wait until Oraxen has finished generating and uploading the pack before players join.

**A new pack version is not sent**

Make sure the new pack has different metadata from the old one. A new hash or UUID allows OneTimePack to identify it as a different pack and let the backend send it.
